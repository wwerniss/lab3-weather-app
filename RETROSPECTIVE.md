#  Ретроспектива (Лабораторна робота №3)

Цей документ містить відповіді на всі 12 пунктів, зазначених у методичці до лабораторної роботи №3, щодо досвіду використання зовнішніх бібліотек і фреймворків у проєкті "Weather App".

---

## 1. Які конкретні задачі планували вирішувати за допомогою цієї бібліотеки?

Метою було створення веб-застосунку, який отримує та відображає дані про поточну погоду і прогноз на кілька днів із зовнішнього API. React використовувався для створення інтерфейсу та логіки компонентів, а Axios — для виконання HTTP-запитів до OpenWeatherMap API.

---

## 2. Чому було обрано саме цю бібліотеку, а не аналоги?

React — один із найпопулярніших фреймворків, має просту компонентну модель, підтримує хуки та добре підходить для динамічних інтерфейсів.  
Axios обрано через зручнішу роботу з промісами, обробкою помилок і компактний синтаксис у порівнянні з `fetch`.  
OpenWeatherMap обрано як якісний, безкоштовний сервіс із повною документацією.

---

## 3. Наскільки просто було отримати, встановити та почати використовувати бібліотеку?

Встановлення React через Vite було дуже простим (npm create vite@latest). Axios встановлюється однією командою (npm install axios). Налаштування проєкту зайняло мінімум часу, і вже через кілька хвилин застосунок працював із базовою структурою.

---

## 4. Наскільки зрозумілою була документація бібліотеки?

Документація React є детальною, структурованою та містить багато прикладів.  
Документація Axios коротка, але зрозуміла.  
OpenWeatherMap має повний опис ендпоінтів і приклади запитів, що значно спростило інтеграцію.

---

## 5. Наскільки було зрозуміло, як саме використовувати бібліотеку, які класи/методи/функції використовувати для вирішення поставлених задач?

У React логіка роботи зі станами через `useState` виявилася простою.  
Метод `axios.get()` має інтуїтивний синтаксис, а результат одразу повертається у форматі JSON.  
При роботі з API було зрозуміло, які саме поля потрібні для відображення.

---

## 6. Наскільки зручно було використовувати бібліотеку, чи не треба було писати багато надлишкового коду?

Ні. React дозволив перевикористовувати компоненти, що зменшило кількість коду. Axios спростив виконання запитів, і майже весь код залишився чистим і коротким. Надлишковий код не виникав.

---

## 7. Наскільки зрозумілою була поведінка класів/методів/функцій з бібліотеки?

Поведінка методів було передбачуваним: axios.get() повертає відповідь у форматі JSON, а помилки чітко вказують, де проблема. React рендерив компоненти строго відповідно до змін станів.

---

## 8. Наскільки зрозумілою була взаємодія між різними класами/методами/функціями цієї бібліотеки, а також взаємодія між бібліотекою та власним кодом?

Взаємодія була організована логічно:  
- окремий модуль `getWeather.js` для API-запитів,  
- окремі компоненти для інтерфейсу,  
- стан і логіка — у головному компоненті `App.jsx`.  

Усі частини працювали разом зрозуміло та передбачувано.

---

## 9. Які проблеми виникали, і як вони були вирішені?

Основна проблема була пов’язана з областю видимості змінної `API_KEY`, через що прогноз не працював. Це було виправлено шляхом винесення ключа у верхній рівень модуля.  
Також виникала необхідність правильно обробляти API-помилки, що було вирішено через коректне кидання текстових помилок.

---

## 10. Позитивні аспекти використання бібліотеки

- React дозволив швидко створити структурований і сучасний UI.  
- Axios значно спростив роботу з API.  
- OpenWeatherMap має просту інтеграцію та багато інформації.  
- Код вийшов чистим, компонентним та масштабованим.

---

## 11. Негативні аспекти використання бібліотеки

- React потребує початкового розуміння хуків і компонентів.  
- OpenWeatherMap повертає прогноз кожні 3 години, що вимагало додаткової обробки.  
- Axios додає зайву залежність (але це мінімальний мінус).

---

## 12. Якби довелось вирішувати аналогічну задачу, але вже враховуючи досвід використання в цій лабораторній роботі, що варто було б робити так само, а що змінити?

Маючи досвід виконання цієї лабораторної роботи, я б зберегла загальну архітектуру проєкту: розділення на компоненти, використання окремого модуля для API-запитів, роботу з Axios та React як основний фреймворк. Такий підхід виявився простим, логічним і зручним. Якщо б довелося ще раз вирішувати подібну задачу, я б розглянула варіант використання додаткових бібліотек. Наприклад, для запитів до API можна було б використати не лише Axios, а й React Query (TanStack Query) — це спростило б кешування, автоматичне оновлення даних і обробку станів завантаження/помилок.
Також для стилізації можна було б використати Tailwind CSS або готову UI-бібліотеку (наприклад, Material UI або Chakra UI), що дозволило б швидше будувати інтерфейс і легше підтримувати консистентний дизайн.
